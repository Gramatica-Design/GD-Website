Schnelleinstieg

Dieses Projekt verbindet Webflow (Design + CMS) mit eigenem JavaScript aus einem GitHub-Repository. Die Skripte werden nicht direkt in Webflow eingebettet, sondern dynamisch über das CDN jsDelivr geladen — direkt von GitHub. Das ermöglicht es, Skripte lokal zu entwickeln und zu testen, ohne den Webflow-Editor anfassen zu müssen.

RessourceLink
GitHub Repositorygithub.com/Gramatica-Design/GD-Website
jsDelivr Purge-Tooljsdelivr.com/tools/purge

Die zwei Script-Loader

Das ist der häufigste Punkt der Verwirrung: Es gibt zwei separate Script-Loader in Webflow, die unabhängig voneinander umgeschaltet werden müssen.

LoaderWo in WebflowLädt
Site Scripts Site Settings → Custom Code → Before </body> cursor.js, nav.js, footer-form.js, text-animation.js — auf allen Seiten
Home Scripts Home Page → Settings → Custom Code → Before </body> process.js, slider.js, list-hover.js, feldlinien.js, magnetic-button.js — nur Home

Lokal entwickeln

  1. In beiden Loadern die Flag auf true setzen und Webflow publishen:
    Site Settings: const isLocal_ss = true;
    Home Page: const isLocal = true;
  2. Live Server in VS Code starten (Port 5500, HTTPS via mkcert — siehe Abschnitt 9.2)
  3. Skript bearbeiten, Browser mit Cmd+Shift+R neu laden

Änderung live schalten

  1. Flags in beiden Loadern wieder auf false setzen, Webflow publishen
  2. Committen und pushen: 
    git add js/dateiname.js
git commit -m "Beschreibung"
git push
  3. jsDelivr-Cache purgen — sonst bleibt die alte Version live.
    Alle URLs auf einmal bei jsdelivr.com/tools/purge eingeben: 
    https://cdn.jsdelivr.net/gh/Gramatica-Design/GD-Website@main/js/cursor.js
https://cdn.jsdelivr.net/gh/Gramatica-Design/GD-Website@main/js/nav.js

02

Script-Ladestrategie

Scripts werden dynamisch zur Laufzeit geladen — nicht als statische <script>-Tags. Dadurch kann zwischen dem jsDelivr-CDN (Produktion) und einem lokalen Live Server (Entwicklung) umgeschaltet werden, ohne den Webflow-Editor zu öffnen.

2.1 Site-weite Scripts

Wo: Site Settings → Custom Code → Before </body>
Lädt vier Scripts auf allen Seiten. Umschaltung via isLocal_ss:

<script>
  const isLocal_ss = false; // <-- hier umschalten: true = localhost, false = CDN
  const localIP_ss = 'https://localhost:5500';
  const base_ss = isLocal_ss
    ? localIP_ss + '/js'
    : 'https://cdn.jsdelivr.net/gh/Gramatica-Design/GD-Website@main/js';
  const files_ss = [
    "cursor.js",
    "nav.js",
    "footer-form.js",
    "text-animation.js"
  ];
  files_ss.forEach((files_ss) => {
    const script_ss = document.createElement('script');
    script_ss.src = `${base_ss}/${files_ss}`;
    document.head.appendChild(script_ss);
  });
</script>

2.2 Home-spezifische Scripts

Wo: Home Page → Settings → Custom Code → Before </body>
Lädt fünf Scripts nur auf der Home-Seite. Umschaltung via isLocal:

<script>
  const isLocal = false; // <-- hier umschalten: true = localhost, false = CDN
  const localIP = 'https://localhost:5500';
  const base = isLocal
    ? localIP + '/js'
    : 'https://cdn.jsdelivr.net/gh/Gramatica-Design/GD-Website@main/js';
  const files = [
    "process.js",
    "slider.js",
    "list-hover.js",
    "feldlinien.js",
    "magnetic-button.js",
  ];
  files.forEach((file) => {
    const script = document.createElement('script');
    script.src = `${base}/${file}`;
    document.head.appendChild(script);
  });
</script>
jsDelivr Cache purgen — nach einem Push auf main, wenn Änderungen nicht erscheinen:
Tool: jsdelivr.com/tools/purge
Beispiel-URL: https://cdn.jsdelivr.net/gh/Gramatica-Design/GD-Website@main/js/slider.js

4.1 Text-Animations-Styles

Wo: Site Settings → Head Code

CSS-Keyframe-Animation für den buchstabenweisen Farbeffekt (text-fade-in) sowie die CSS-Custom-Property für den Ease.

Keyframe-Logik: Jeder Buchstabe animiert von inherit → Blau (#66b2ef) → Grau (--_semantics---text_alternate) → Endfarbe. Die Endfarbe wird über --char-final-color gesteuert: Standard ist Schwarz; mit dem [data-accent]-Attribut auf dem Container bleibt der Buchstabe Blau.

:root {
  --ease-out-power2: cubic-bezier(0.215, 0.61, 0.355, 1);
}

/* Buchstaben die nach der Animation blau bleiben: [data-accent] auf den Container setzen */
[data-accent] {
  --char-final-color: var(--_semantics---text_blue);
}

[use-letter-spacing] .split-char:first-child {
  letter-spacing: -0.06em;
}

h1 {
  font-kerning: none;
}

[data-text-fade-in] .split-line .split-char.animate {
  animation: text-fade-in 0.3s var(--ease-out-power2) both;
}

.text-big .split-char.animate {
  animation: text-fade-in 0.3s var(--ease-out-power2) both;
}

@keyframes text-fade-in {
  0%   { color: inherit; }
  10%  { color: #66b2ef; }
  20%  { color: #66b2ef; }
  30%  { color: var(--_semantics---text_alternate); }
  60%  { color: var(--_semantics---text_alternate); }
  65%  { color: var(--char-final-color, var(--_semantics---text)); }
  to   { color: var(--char-final-color, var(--_semantics---text)); }
}

4.4 w-designmode

Wo: Site Settings oder Global Embed

Überschreibt Styles nur im Webflow Designer (html.wf-design-mode ist aktiv). Elemente, die per JavaScript auf height: 0; opacity: 0 gesetzt werden, sind im Designer damit bearbeitbar.

5.3.1 submit_embed — Custom Element footer_form_submit-btn

Wo:HTML Embed innerhalb Custom Element footer_form_submit-btn
Klasse:submit_embed

Der Submit-Button des Footer-Formulars ist ein Custom Element (footer_form_submit-btn) und enthält einen HTML Embed.

    btn.setAttribute('type', 'submit');

</script>

Scroll-Lock beim Seitenaufruf

Beim Laden bei scrollY === 0 wird das Scrollen für die Dauer der Entrance-Animation gesperrt (wheel/touchmove/keydown Events abgefangen). Dadurch kollidiert die Entrance-Animation nicht mit der Magnet-ScrollTrigger-Animation. Der Scrollbalken bleibt sichtbar (kein overflow: hidden), um Layout-Shifts zu vermeiden.

Reload-Handling

Wenn scrollY > 0 beim Laden (Reload bei gescrollter Position): Entrance-Animation wird übersprungen, Buchstaben werden sofort eingeblendet via gsap.set(chars, { y: 0, opacity: 1 }).

Safari-Fix für kursive Buchstaben

Kursive .split-char-Elemente innerhalb von .split-line.accent bekommen padding: 0.1em; margin: -0.1em via Inline-Style (nicht CSS), damit der kursive Überhang nicht an der inline-block-Box abgeschnitten wird.

Abhängigkeiten:GSAP, ScrollTrigger, ScrollToPlugin

06

Cursor-System

Das Cursor-System besteht aus drei voneinander abhängigen Teilen:

6.1 Webflow-Komponente: Cursor

Eine wiederverwendbare Webflow-Komponente, die das .cursor-Div enthält. In der Komponente werden Grösse und CSS-Transitions definiert (Webflow-Styles) und der cursor_embed eingebettet (HTML Embed).

6.2 cursor.js — Bewegung

  • Liest mousemove-Events
  • Bewegt .cursor via gsap.quickTo (x und y) mit duration: 0.15
  • gsap.set(cursor, { xPercent: -50, yPercent: -50 }) zentriert das Element auf der Mausposition
  • Läuft nur bei (pointer: fine) — kein Effekt auf Touch-Geräten

6.3 cursor_embed CSS — Verhalten

  • * { cursor: none } — Standard-Cursor überall ausblenden
  • :has()-Selektoren weiten den Cursor auf 3rem aus bei Hover über interaktive Elemente
  • @media (pointer: coarse) stellt Standard-Cursor wieder her und versteckt .cursor

6.4 Cursor auf neuen Elementen vergrössern

Neuen Selektor in cursor_embed in der Cursor-Komponente ergänzen:

body:has(.mein-element:hover) .cursor {
  width: 3rem;
  height: 3rem;
}

Bekannte Probleme

11.1 Navigation Scroll-Präzision (erster Klick)

Status:Offen — akzeptabler Kompromiss
Betrifft:nav.js · Sections Work & About

Symptom: Beim ersten Klick auf „Work" oder „About" in der Navigation landet die Seite ~107px zu tief. Beim zweiten Klick auf denselben oder einen anderen Link funktioniert die Navigation präzise.

Root Cause

Zwei überlagerte Probleme treffen beim ersten Klick zusammen:

  1. Webflows eigener Scroll-Handler berechnet die Zielposition einmalig beim Klick als feste Pixelzahl und scrollt dorthin. Ein parallel gestarteter eigener Handler kommt dabei zu spät.
  2. Dynamische Dokumenthöhe: Die Process-Section expandiert ihre Steps über GSAP-ScrollTrigger-Animationen. Beim Durchscrollen wächst die Dokumenthöhe — die vorab berechnete Zielposition stimmt dadurch nicht mehr.

Beim zweiten Klick hat sich Webflow neu kalibriert und die berechnete Position stimmt zufällig mit der tatsächlichen überein.

Versuche

AnsatzBeschreibungErgebnis
rAF chaseScroll getBoundingClientRect().top kontinuierlich messen, mit window.scrollBy nachkorrigieren Funktioniert beim zweiten Klick, nicht beim ersten
Capture-Phase Interception Event-Listener mit { capture: true } + e.stopPropagation() um Webflows Handler zu verhindern Hat das Problem ebenfalls nicht gelöst

Mögliche zukünftige Ansätze

  • ScrollTrigger.refresh() vor dem Scroll aufrufen, damit alle Positionen aktuell sind
  • process.js so anpassen, dass alle Steps beim Initialisieren sofort in ihren finalen Zustand gebracht werden (statt animiert beim Scrollen)
  • Webflows Smooth-Scroll über Webflow.destroy() vollständig deaktivieren und durch eigene Logik ersetzen
