BUG-01

Navigation Scroll-Präzision

Status: Offen
Betrifft:nav.js
Sections:Work, About
Priorität:Niedrig

Symptom

Beim ersten Klick auf „Work" oder „About" in der Navigation landet die Seite ~107px zu tief. Ab dem zweiten Klick funktioniert die Navigation korrekt.

Root Cause

Zwei Probleme treffen beim ersten Klick zusammen:

  1. Webflows eigener Scroll-Handler berechnet die Zielposition einmalig beim Klick als feste Pixelzahl. Ein parallel laufender eigener Handler kommt zu spät — Webflow hat bereits gescrollt.
  2. Dynamische Dokumenthöhe: Die Process-Section expandiert ihre Steps via GSAP-ScrollTrigger-Animationen. Beim Durchscrollen wächst die Dokumenthöhe — die vorab berechnete Zielposition stimmt dadurch nicht mehr.

Beim zweiten Klick hat sich Webflow intern neu kalibriert und die berechnete Position stimmt zufällig mit der tatsächlichen überein.

Lösungsversuche

AnsatzBeschreibungErgebnis
rAF chaseScroll getBoundingClientRect().top auf jedem Frame neu messen, mit window.scrollBy nachkorrigieren Funktioniert beim zweiten Klick, nicht beim ersten
Capture-Phase Interception Event-Listener mit { capture: true } + e.stopPropagation() um Webflows Handler zu blockieren Hat das Problem ebenfalls nicht gelöst

Mögliche zukünftige Ansätze

  • ScrollTrigger.refresh() vor dem Scroll aufrufen, damit alle Trigger-Positionen aktuell sind
  • process.js so anpassen, dass alle Steps beim Initialisieren sofort in ihren finalen Zustand gebracht werden — keine Höhenänderung während des Scrollens
  • Webflows Smooth-Scroll via Webflow.destroy() vollständig deaktivieren und durch eigene Logik ersetzen
BUG-02

Custom Cursor Re-Entry

Status: Teilweise behoben
Betrifft:cursor.js
Fix in:Commit 6c0642e
Priorität:Niedrig

Symptom

Wenn man vom Browser zu einer anderen Applikation wechselt und dann zurückkommt, bleibt der Custom Cursor an seiner letzten Position stehen. Beim Zurückwechseln in den Browser wird sichtbar, wie der Cursor von dieser alten Position zur neuen Maus-Position animiert — ein sichtbarer Schwenk über den Bildschirm.

Root Cause

Browser feuern keine mousemove-Events wenn das Fenster nicht fokussiert ist — das ist eine Browser-Sicherheitsbeschränkung, die sich nicht umgehen lässt. Während der Nutzer in einer anderen App arbeitet, aktualisiert sich die Cursor-Position nicht. Beim Zurückkehren animiert gsap.quickTo mit seiner 0.15s-Duration vom alten zum neuen Ort.

Angewandter Fix

mouseenter auf document erkennt den Moment des Wiedereintritts in den Browser. gsap.set (ohne Animation) setzt den Cursor sofort an die korrekte Position — kein Schwenk, kein Sprung.

document.addEventListener("mouseenter", (e) => {
  gsap.set(cursor, { x: e.clientX, y: e.clientY });
});
Verbleibende Einschränkung: Wenn man direkt in das Browserfenster klickt (ohne vorher zu hovern), gibt es noch einen kurzen Sprung — mouseenter feuert nicht bei direkten Klicks ohne Hover-Phase.

Empfehlung

Custom Cursor langfristig entfernen. Der Custom Cursor ist ein 3-teiliges System (Webflow-Komponente + cursor.js + cursor_embed CSS), das inherente Browser-Einschränkungen hat, die sich technisch nicht vollständig lösen lassen. Der System-Cursor bietet auf allen Geräten eine bessere, konsistentere UX ohne Implementierungsaufwand. Hover-Effekte auf interaktiven Elementen lassen sich direkt auf den Elementen implementieren, ohne einen Custom Cursor zu benötigen.

Schritte zum Entfernen des Custom Cursors

  1. In Webflow: Cursor-Komponente aus allen Seiten entfernen
  2. cursor.js aus dem Site-Script-Loader entfernen
  3. In cursor_embed: * { cursor: none; } entfernen — damit ist der System-Cursor wieder aktiv
  4. cursor_embed kann danach vollständig gelöscht werden
  5. Alle body:has(…:hover) .cursor-Selektoren aus dem CSS entfernen
